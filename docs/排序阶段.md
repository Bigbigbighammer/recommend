# 排序阶段开发文档

## 一、架构设计

### 1.1 管线位置

```
RecallStage (≤100 候选) → RankingStage (本阶段) → RerankingStage
                                │
                    DeepFM CTR 预估 → 按分数降序 → Top-20
```

排序阶段不依赖召回的具体实现——只要上游产出 `List<RecallItem>`（含 `movieId`），即可对候选集打分排序。

### 1.2 DeepFM 模型

```
user_id, gender, age, occupation, zip_code, movie_id, genres, is_adult
        │                                    │
        └──────────┬─────────────────────────┘
                   │
         ┌─────────▼─────────┐
         │  Embedding (16维)  │  ← FM Linear + FM Interaction + DNN 共享
         └─────────┬─────────┘
                   │
    ┌──────────────┼──────────────┐
    ▼              ▼              ▼
FM Linear      FM Interaction    Deep (MLP [64,32,16] + ReLU + Dropout + BN)
    │              │              │
    └──────────────┴──────────────┘
                   │
               Sigmoid → CTR ∈ (0,1)
```

| 组件 | 作用 |
|------|------|
| FM Linear | 一阶特征权重：`Σ w_i·x_i + w_0` |
| FM Interaction | 二阶特征交叉：`0.5·Σ[(Σv_i)² - Σv_i²]` |
| Deep | 高阶交叉：MLP 自动学习任意阶特征组合 |

### 1.3 特征字段

| 特征 | 取值 | 编码 |
|------|------|------|
| user_id | 1~6040 | LabelEncoder → Embedding |
| gender | M / F | 同上 |
| age | Under 18, 18-24, ... (7种) | 同上 |
| occupation | K-12 student, engineer, ... (21种) | 同上 |
| zip_code | 48067, ... (2960种) | 同上 |
| movie_id | 1~3883 | 同上 |
| genres | 首类型字符串 (18种) | 同上 |
| is_adult | 0 / 1 | 同上 |

### 1.4 文件结构

```
scripts/
├── preprocess_ranking.py    ← 数据预处理（PG → 编码 → 负采样 → npz）
├── train_ranking.py         ← DeepFM 训练 + 模型导出
└── eval_ranking.py          ← 离线评估（AUC/GAUC/NDCG/Prec/Recall/HR）

inference-service/
├── main.py                  ← [已修改] /api/predict/ranking 用真实模型
├── model_ranking.py         ← DeepFM 模型定义 + RankingPredictor
└── model_weights/
    ├── deepfm_model.pt      ← 训练产物：模型权重 + 元数据
    ├── ranking_encoders.pkl ← 训练产物：LabelEncoder 字典
    └── ranking_feature_dims.pkl ← 训练产物：特征维度

scripts/output_ranking/      ← 预处理产物（train/val/test .npz）
```

## 二、运行指令

### 2.1 环境准备

```bash
pip install scikit-learn numpy pandas psycopg2-binary \
  -i https://pypi.tuna.tsinghua.edu.cn/simple

pip install torch --index-url https://download.pytorch.org/whl/cpu \
  --extra-index-url https://pypi.tuna.tsinghua.edu.cn/simple
```

### 2.2 预处理

```bash
# 从 PG 读取数据 → 正负样本构造 → LabelEncoder 编码 → npz 输出
python3 scripts/preprocess_ranking.py --neg-ratio 4 --pos-threshold 4.0
```

正样本 = 评分 ≥ 4，负样本 = 评分 < 4 + 随机负采样（每正样本 4 个）。按用户 8:1:1 划分。

### 2.3 训练

```bash
python3 scripts/train_ranking.py --epochs 30 --lr 1e-3
```

| 超参 | 值 |
|------|-----|
| Embedding 维度 | 16 |
| DNN 隐层 | [64, 32, 16] |
| Dropout | 0.2 |
| Batch Size | 512 |
| Optimizer | Adam (lr=1e-3, weight_decay=1e-5) |
| 早停 | val_loss 连续 3 轮不降 |
| 学习率衰减 | ReduceLROnPlateau (factor=0.5, patience=2) |

### 2.4 评估

```bash
python3 scripts/eval_ranking.py --output metrics.json
```

### 2.5 启动推理服务

```bash
cd inference-service
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8080
```

验证：

```bash
curl -X POST http://localhost:8080/api/predict/ranking \
  -H "Content-Type: application/json" \
  -d '{
    "user_features": {"userId": 1, "gender": "F"},
    "item_features": [
      {"movie_id": 6, "genres": "Action", "isAdult": 0},
      {"movie_id": 32, "genres": "Drama", "isAdult": 0}
    ]
  }'
```

### 3.3 测试集指标与 Baseline 对比

**Baseline 是什么**：在模型评估中，baseline 是用来证明复杂模型"确实有价值"的参照物。如果 DeepFM 不比最简单的模型好，那复杂设计就没有意义。

| Baseline | 方法 | 回答什么问题 |
|----------|------|-------------|
| **Random** | 随机打分 | 数据有没有标签泄漏？AUC > 0.5 才说明模型学到了东西 |
| **Popularity** | 按物品在训练集中出现频率排序 | 个性化有没有价值？DeepFM 必须超过"给所有人推热门" |

实测结果：

```
======================================================================
  Ranking Model Comparison
======================================================================
  Model               AUC     GAUC  LogLoss    NDCG@5   NDCG@10   NDCG@20
  -------------- -------- -------- --------  --------  --------  --------
  DeepFM           0.8575   0.8651   0.3588    0.8477    0.8259    0.7848
  Popularity       0.8617   0.8653   0.3467    0.8624    0.8412    0.7986
  Random           0.5015   0.5015   0.9979    0.2087    0.1998    0.2013
----------------------------------------------------------------------
  DeepFM vs Popularity:  AUC -0.0042,  GAUC -0.0002,  NDCG@10 -0.0153
======================================================================
```

**关键发现**：DeepFM 略**低于** Popularity baseline。这说明现阶段个性化排序并未带来增益，"推荐热门"反而效果更好。

原因分析：
1. 大量 test 用户的 `user_id` 在训练集中未出现过，LabelEncoder 统一映射为 0（"未知用户"），模型无法区分不同新用户
2. 物品特征太少（仅 genres + is_adult），缺少 rating_count、avg_rating 等统计特征
3. 正负样本定义（评分 ≥ 4 = 正样本）天然偏向热门电影——热门电影评分多且高

改进方向：
- 增加用户侧统计特征（历史平均评分、活跃度），让未知用户也有差异化的表示
- 增加物品侧统计特征（rating_count, avg_rating），让 DeepFM 至少能学到和 Popularity 一样的信息
- 改用相对阈值（rating ≥ 用户平均分）替代绝对阈值，减少热门偏差

## 五、工作说明

### 5.1 已完成

- **数据处理**：从 PG 三表（ratings/users/movies）抽取、合并、负采样、LabelEncoder 编码、数据集划分
- **模型训练**：PyTorch DeepFM（FM Linear + FM Interaction + DNN），早停 + 学习率衰减
- **推理部署**：`/api/predict/ranking` 端点加载真实模型，模型不存在时自动降级为随机分数
- **离线评估**：AUC / GAUC / LogLoss / NDCG@K / Precision@K / Recall@K / HitRate@K

### 5.2 与上下游接口

- **上游（召回）**：接收 `List<RecallItem>`（需含 `movieId`），不依赖召回分数
- **下游（重排）**：输出按 CTR 降序的 Top-20，供 Genre/Decade 打散
- **协议**：`POST /api/predict/ranking` 请求/响应格式已由 Java 侧定义，不可修改

### 5.3 已知限制与优化方向

| 问题 | 当前方案 | 优化方向 |
|------|---------|---------|
| genres 多值特征 | 仅取首类型 | 拆成多二值特征 或 多头注意力 pooling |
| 用户冷启动 | 新用户编码为 0 | 引入用户画像侧的统计特征（avg_rating, 活跃度） |
| 物品特征少 | 仅 genres + is_adult | 加入 year, rating_count, imdb_rating 等 |
| Recall 偏低 | 测试集正样本密度高（~161/用户） | 实际系统召回先过滤到 100 候选，Recall 会显著提高 |

## 七、优化迭代记录

### 7.0 初始状态（基线）

**特征**：8 个（user_id, gender, age, occupation, zip_code, movie_id, genres, is_adult）
**标签**：rating >= 4.0（绝对阈值）

```
Model         AUC     GAUC   NDCG@5   NDCG@10   NDCG@20
DeepFM       0.8575  0.8651   0.8477    0.8259    0.7848
Popularity   0.8617  0.8653   0.8624    0.8412    0.7986
DeepFM vs Pop  -0.0042 -0.0002  -0.0147   -0.0153   -0.0138
```

**问题**：DeepFM 全面落后于 Popularity baseline，个性化排序未带来增益。

---

### 7.1 Step 1：增加物品统计特征

**新增特征**（3 个）：year, avg_rating_bin, rating_count_bin
**总特征数**：11
**分桶策略**：
- year → 年代（pre-1950, 1950s, ..., 2010s）
- avg_rating → 0.5 分桶（1.0, 1.5, ..., 10.0）
- rating_count → 对数分桶（0, 1-5, 6-20, 21-50, 51-100, 101-200, 201-500, 501+）

**修改文件**：
- `scripts/preprocess_ranking.py`：ITEM_FEATURES 扩展 + 分桶函数 + SQL 查询增加列
- `scripts/train_ranking.py`：ITEM_FEATURES 同步
- `scripts/eval_ranking.py`：ITEM_FEATURES 同步
- `inference-service/model_ranking.py`：feature_names + item_map 扩展
- `ItemFeatures.java`：新增 avgRating, ratingCount 字段
- `RankingStage.java`：fetchItemFeatures 读取新字段
- `DeepFMRankingStrategy.java`：传递 year, avgRating, ratingCount

```
Model         AUC     GAUC   NDCG@5   NDCG@10   NDCG@20
DeepFM       0.8589  0.8663   0.8345    0.8121    0.7755
Popularity   0.8617  0.8653   0.8624    0.8412    0.7986
DeepFM vs Pop  -0.0029 +0.0010  -0.0279   -0.0291   -0.0231
```

**结论**：GAUC 首次超越 Popularity（+0.0010），AUC 差距缩小。NDCG 下降——物品统计特征增强了 Popularity 信号的权重，但 top-K 排序变差。需要用户侧特征来平衡。

---

### 7.2 Step 2：增加用户统计特征

**新增特征**（3 个）：user_avg_rating_bin, user_rating_count_bin, user_active_days_bin
**总特征数**：14
**计算方式**：从 ratings 表按 user_id 聚合（avg, count, max/min timestamp）
**分桶策略**：avg_rating 与物品侧共用分桶，rating_count 对数分桶，active_days 分桶（0, 1-7, 8-30, 31-90, 91-180, 181-365, 365+）

**修改文件**：
- `scripts/preprocess_ranking.py`：USER_FEATURES 扩展 + 用户聚合统计 + 分桶合并
- `scripts/train_ranking.py`：USER_FEATURES 同步
- `scripts/eval_ranking.py`：USER_FEATURES 同步
- `inference-service/model_ranking.py`：feature_names + user_map + 分桶逻辑
- `RatingMapper.java`：新增 selectStatsByUser 查询
- `UserFeatureEnrichmentStage.java`：注入 RatingMapper，计算并填充用户统计到 enriched map

```
Model         AUC     GAUC   NDCG@5   NDCG@10   NDCG@20
DeepFM       0.8675  0.8690   0.8429    0.8228    0.7870
Popularity   0.8617  0.8653   0.8624    0.8412    0.7986
DeepFM vs Pop  +0.0058 +0.0037  -0.0195   -0.0184   -0.0116
```

**结论**：AUC/GAUC 大幅领先 Popularity（+0.0058/+0.0037），NDCG 差距显著缩小。用户统计特征让 unseen user 有了差异化表示。但 NDCG 仍落后——根因在"热门=高分"的标签偏差。

**收敛变化**：模型收敛变慢（最优 epoch 从 4 推迟到 8），因为新增特征需要更多训练来学习交互。

---

### 7.3 Step 3：相对阈值标签

**改动**：标签从 `rating >= 4.0` 改为 `rating >= 用户平均分`
**效果**：正样本占比从 94.4% 降到 54.5%，类别更均衡
**意义**：推荐任务从"预测高分物品"变为"预测对特定用户超出预期的物品"，从根本上消除热门偏差

**修改文件**：
- `scripts/preprocess_ranking.py`：新增 --relative-threshold 参数，修改标签计算逻辑

```
Model         AUC     GAUC   NDCG@5   NDCG@10   NDCG@20
DeepFM       0.8603  0.8639   0.7338    0.7063    0.6828
Popularity   0.8514  0.8579   0.7391    0.7062    0.6806
DeepFM vs Pop  +0.0090 +0.0060  -0.0053   +0.0001   +0.0022
```

**结论**：NDCG@10 追平 Popularity，NDCG@20 首次超越！AUC/GAUC 领先幅度进一步扩大至 +0.009/+0.006。

> 注意：相对阈值改变了标签定义，绝对值不可与 Step 1-2 直接对比。DeepFM vs Popularity 的**差值**才是核心指标。

---

### 7.4 Step 4：Multi-hot Genres

**改动**：将单一 `genres` 特征（仅取首类型）替换为 18 个 `genre_<Name>` 二值特征
**genres 列表**：Action, Adventure, Animation, Childrens, Comedy, Crime, Documentary, Drama, Fantasy, FilmNoir, Horror, Musical, Mystery, Romance, SciFi, Thriller, War, Western
**总特征数**：14 - 1 + 18 = 31

**修改文件**：
- `scripts/preprocess_ranking.py`：动态提取所有 genre，生成二值特征列，feature_dims 保存 all_features 和 genre_features
- `scripts/train_ranking.py`：ALL_FEATURES 动态加载（从 feature_dims.pkl），DeepFM 存储 feature_names
- `scripts/eval_ranking.py`：ALL_FEATURES 动态加载
- `inference-service/model_ranking.py`：feature_names 从 checkpoint 加载，user_field_names 动态检测
- `DeepFMRankingStrategy.java`：用 genresList() 生成 genre_<Name>=1 多热编码

**结果**：

```
Model         AUC     GAUC   NDCG@5   NDCG@10   NDCG@20
DeepFM       0.8564  0.8594   0.7345    0.7072    0.6788
Popularity   0.8514  0.8579   0.7391    0.7062    0.6806
DeepFM vs Pop  +0.0050 +0.0015  -0.0046   +0.0010   -0.0018
```

**结论**：Multi-hot genres 仍保持正向收益（AUC +0.005 vs Pop），但相比 Step 3 全面退步（AUC +0.009→+0.005, GAUC +0.006→+0.0015）。18 个二值特征用 16 维 embedding 过于浪费（576 额外参数 vs 仅 0/1 信息），FM 二阶交叉引入噪声，训练时间暴增 3 倍（165 min vs 57 min）。

**后续方向**：改用低维 genre embedding（如 4 维）或 avg pooling 聚合多 genre 为单一向量，而非独立二值 embedding。

**改动**：回退 multi-hot genres，恢复单 genres 特征，新增 preferred_genre_match 交叉特征
**特征数**：15（8 用户 + 7 物品）
**修改文件**：
- `preprocess_ranking.py`：移除 genre 二值特征，恢复单 genres，添加 preferred_genre_match 计算
- `model_ranking.py`：添加 preferredGenreMatch 映射
- `DeepFMRankingStrategy.java`：计算 preferredGenreMatch 交叉特征

**结论**：回退成功（训练时间 47 min，接近 Step 3 的 57 min）。preferred_genre_match 在 ML1M 上无效（6040/6041 用户 preferred_genres 为 NULL），特征仅 1 个类别。

### 7.6 Step 6：P1 交叉特征 + IMDb 特征

**新增特征**（3 个）：rating_deviation_bin, imdb_rating_bin, imdb_votes_bin
**总特征数**：17（8 用户 + 9 物品）

| 特征 | 类别 | 含义 |
|------|:----:|------|
| rating_deviation_bin | 35 | user_avg_rating - movie_avg_rating，0.5 分桶 |
| imdb_rating_bin | 16 | IMDb 评分，0.5 分桶 |
| imdb_votes_bin | 7 | IMDb 投票数，对数分桶 |

**修改文件**：
- `preprocess_ranking.py`：ITEM_FEATURES 扩展 + 分桶 + neg_samples更新
- `model_ranking.py`：binning + item_map
- `ItemFeatures.java`：新增 imdbRating, imdbVotes
- `RankingStage.java`：读取 IMDb 字段
- `DeepFMRankingStrategy.java`：计算 ratingDeviation + 传递 IMDb 字段

**结果（16-dim embedding, lr=1e-3）**：

```
Model         AUC     GAUC   NDCG@5   NDCG@10   NDCG@20
DeepFM       0.8573  0.8616   0.7393    0.7058    0.6876
Popularity   0.8514  0.8579   0.7391    0.7062    0.6806
DeepFM vs Pop  +0.0060 +0.0037  +0.0002   -0.0004   +0.0070

**结论**：17 特征在 16 维 embedding 下不如 14 特征的 Step 3（AUC +0.006 vs +0.009）。每增加一个特征，16 维 embedding 的表征能力被进一步稀释。NDCG@20 有改善（+0.0070 vs Step 3 的 +0.0022），说明外部质量信号对长列表排序有帮助。

Popularity   0.8514  0.8579   0.7391    0.7062    0.6806
DeepFM vs Pop  +0.0025 +0.0010  -0.0048   +0.0003   -0.0010
```

**结论**：32 维 + lr=5e-4 在 CPU 上收敛过慢（107 min，最优 epoch 28），结果反而最差（AUC 仅 +0.0025）。在 CPU 约束下，16 维 embedding + lr=1e-3 是性价比最优配置。

### 7.8 最终结论

| 阶段 | 特征 | 标签 | Embed | 训练 | AUC vs Pop | GAUC vs Pop | NDCG@10 vs Pop |
|------|:----:|------|:-----:|------|-----------|-------------|----------------|

| Step2 +用户统计 | 14 | 绝对 | 16 | 58m | +0.0058 | +0.0037 | -0.0184 |
| **Step3 +相对阈值** | **14** | **相对** | **16** | **57m** | **+0.0090** | **+0.0060** | **+0.0001** |
| Step4 +multi-hot | 31 | 相对 | 16 | 165m | +0.0050 | +0.0015 | +0.0010 |

| Step7 超参调优 | 17 | 相对 | 32 | 107m | +0.0025 | +0.0010 | +0.0003 |

**最优生产配置：Step 3**（14 特征、相对阈值、16-dim embedding、lr=1e-3）

**核心发现**：
1. **16 维 embedding 的饱和点在 ~14 特征**：超过此数，新增特征的 embedding 参数挤占既有特征的表征空间，导致整体指标下降
2. **相对阈值标签是最大单次提升**：消除了"热门=高分"标签偏差，NDCG@10 从 -0.018 追至 +0.000
3. **在 CPU 约束下，大模型反而不如小模型**：32 维 embedding 收敛过慢，训练 107 分钟仍不如 16 维 57 分钟的结果
4. **外部特征（IMDb）+ 交叉特征需要更大的 embedding 容量**：在 GPU 或更长训练时间下可能超越 Step 3

**后续方向**：
- 用 GPU 训练 32 维大模型（预估 AUC 可达 +0.015 以上）
- 多数据集交叉验证（ML-10K, 100K）
- 尝试 DCN / xDeepFM 等显式交叉网络替代 DeepFM

**最优方案：Step 3（物品统计 + 用户统计 + 相对阈值）**，14 个特征，AUC 领先 +0.009，NDCG@10 追平 Popularity，训练 57 分钟。此方案在 AUC/GAUC 上大幅领先 Popularity，NDCG 首次不再落后，是生产部署推荐方案。
