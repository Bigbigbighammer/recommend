# 召回阶段开发指引

## 一、召回在管线中的位置

```
POST /api/recommendations/recommend
         │
         ▼
  1. UserFeatureEnrichmentStage    ← Redis 取用户画像 + 历史记录
  2. ColdStartDetectionStage       ← 历史 < 5 条 → 冷启动（跳过召回）
         │
         │  温暖用户 (非冷启动)
         ▼
  3. RecallStage  ← 你要完成的阶段
         │
         │  输出: List<RecallItem> (最多100个候选)
         ▼
  4. RankingStage   ← 排序阶段，从 ctx.recallCandidates() 读取你的产出
  5. RerankingStage
```

**你只需要保证 `recall()` 方法返回 `Mono<List<RecallItem>>`**，PipelineContext 自动将结果传递给排序阶段，不需要你做任何对接。

## 三、你需要创建的文件（3个）

### 3.1 `inference-service/model.py` — 模型定义 + 特征编码

```
inference-service/
├── main.py              ← 修改：加载模型替代 mock
├── model.py             ← 新建：模型定义 + 特征编码器
├── requirements.txt     ← 修改：添加 torch/numpy 等依赖
└── model_weights/       ← 新建目录：存放训练好的权重
```

**内容**：

- 特征列定义（用户侧 + 物品侧），对齐 Java 侧传递的字段名
- 特征编码器（LabelEncoder / OrdinalEncoder），对类别特征做编码
- YouTubeDNN 模型类：
  - **User Tower**：用户画像特征 Embedding + 历史电影 ID 序列 Embedding → 均值池化 → MLP → 16 维 user vector
  - **Item Tower**：物品特征 Embedding → MLP → 16 维 item vector
- 模型加载/保存方法

**关键约束**：user vector 输出维度必须与 `item_emb.npy` 列数一致（当前 16 维），否则 Java 侧 `ItemEmbeddingStore.topK()` 会报维度不匹配。

### 3.2 `scripts/preprocess.py` — 数据预处理

从 PG 三张表抽取、清洗、编码、划分数据集：

```
输入:
  PG ratings 表  → 用户-电影交互（正样本）
  PG users 表    → 用户画像 (gender, age, occupation, zip_code)
  PG movies 表   → 电影属性 (genres, year)

处理:
  1. 读取三张表，join 成宽表
  2. 过滤交互数过少的用户/电影（如 < 5 条）
  3. 划分用户历史序列：按 timestamp 排序，每个用户最后 N 条交互作为测试集
  4. 对类别特征做 LabelEncoder（保存编码器供推理使用）
  5. 负采样：为每个正样本采样 N 个负样本（未交互的电影）
  6. 输出 train/val/test 的 npz 或 pickle

输出:
  scripts/output/
  ├── train_data.npz       ← 训练样本
  ├── val_data.npz         ← 验证样本
  ├── test_data.npz        ← 测试样本
  └── encoders.pkl         ← LabelEncoder 字典 {列名: 编码器}
```

### 3.3 `scripts/train_youtubednn.py` — 训练 + 导出

**流程**：

```
preprocess 输出
    │
    ▼
构造训练样本   ← (user_features, hist_movie_ids, pos_item_id, neg_item_ids)
    │
    ▼
训练 YouTubeDNN ← sampled softmax loss / cross-entropy
    │
    ├──→ 导出 item tower → data/item_emb.npy + data/movie_ids.npy
    ├──→ 导出 user tower → inference-service/model_weights/user_tower.pt
    └──→ 复制 encoders  → inference-service/model_weights/encoders.pkl
```

**产物**（必须生成以下文件，Java 侧启动时读取）：

| 产物 | 路径 | 用途 |
|------|------|------|
| item_emb.npy | `data/item_emb.npy` | 所有电影向量，ItemEmbeddingStore 启动加载 |
| movie_ids.npy | `data/movie_ids.npy` | 与 embedding 行对应的电影 ID |
| user tower 权重 | `inference-service/model_weights/` | main.py 启动加载 |
| 特征编码器 | `inference-service/model_weights/encoders.pkl` | 推理时对输入做编码 |

## 五、与 Java 侧的接口约束

### 5.1 你的产出如何在 Java 侧被消费

```
POST /api/predict/recall
  │
  │  Python 返回 user_vector (16维 float 数组)
  ▼
YouTubeDNNRecallStrategy
  │
  │  调用 embeddingStore.topK(userVector, topK=100, seen, "youtubednn")
  ▼
ItemEmbeddingStore.topK()
  │
  │  遍历所有 item embedding，做点积，MinHeap 取 Top-100
  ▼
List<RecallItem>  ← [{movieId: 47, score: 0.92, recallType: "youtubednn"}, ...]
  │
  │  与 ItemEmbeddingStrategy + UserPreferenceStrategy 的结果蛇形合并去重
  ▼
PipelineContext.recallCandidates  ← 最多 100 个，传给 RankingStage
```

### 5.2 维度约束

- `user_vector` 长度 == `item_emb.npy` 列数 == 16
- 如果改维度，同步修改：
  - `train_youtubednn.py` 中的 `EMBEDDING_DIM` 参数
  - `generate_embeddings.py` 中的 `EMBEDDING_DIM`（如果有）
  - Java 侧 `ItemEmbeddingStore` 会自动适配，无需改

### 5.3 特征字段对齐

Java 侧 `UserFeatureEnrichmentStage` 从 Redis 读取并在 `PipelineContext.userFeatures` 中填充的字段：

| 字段名 | 类型 | 来源 | 说明 |
|--------|------|------|------|
| `userId` | Long | Redis `user:{id}:profile` | 用户 ID |
| `gender` | String | Redis `user:{id}:profile` | "M" / "F" / "O" |
| `age` | String | Redis `user:{id}:profile` | "Under 18" / "18-24" / "25-34" / ... |
| `occupation` | String | Redis `user:{id}:profile` | "K-12 student" / "engineer" / ... |
| `zipCode` | String | Redis `user:{id}:profile` | "48067" |
| `histMovieIds` | List\<Long\> | Redis `user:{id}:history` | 历史观看电影 ID，按时间排序 |
| `preferredGenres` | String | Redis `user:{id}:profile` | "Action,Drama" |
| `frequent_genres` | String | Redis `user:{id}:profile` | 由 ingest_data.py 预计算的热门类型 |

你需要在 `model.py` 的特征编码器中处理这些字段名（snake_case）。

## 六、数据预处理详细说明

### 6.1 现有数据

`ingest_data.py` 已经将数据导入到位：

| 表 | 行数（估计） | 关键列 |
|----|-------------|--------|
| `users` | ~6,040 | user_id, gender, age, occupation, zip_code |
| `movies` | ~3,883 | movie_id, title, year, genres[], imdb_rating |
| `ratings` | ~1,000,000 | user_id, movie_id, rating, timestamp |

用户画像中的 `gender`/`age`/`occupation`/`zip_code` 来自 MovieLens 1M 数据集，取值如下：

| 特征 | 示例值 |
|------|--------|
| gender | "M", "F" |
| age | "Under 18", "18-24", "25-34", "35-44", "45-49", "50-55", "56+" |
| occupation | "K-12 student", "engineer", "executive/managerial", ... （共 21 种） |
| zip_code | "48067" 等，无需编码可按 hash 处理 |

### 6.2 预处理要做的事

1. **读取**：从 PG 读取 ratings/users/movies 三表
2. **过滤**：去掉交互数 < 5 的用户和电影（冷启动用户走冷启动管线，不走召回）
3. **构造序列**：按 timestamp 对每个用户的交互排序，前 N-1 条为历史，最后 1 条为待预测
4. **特征编码**：对类别特征做 LabelEncoder，数值特征做标准化（MinMax/StandardScaler）。保存编码器为 pkl。
5. **负采样**：为每个正样本随机采样 4~10 个用户未交互过的电影作为负样本
6. **划分数据集**：按用户划分 train/val/test（如 8:1:1），保证同一用户的交互不会跨越数据集
7. **输出**：序列化为 npz 或 pickle 供训练脚本读取

## 七、模型设计参考

### YouTubeDNN 结构

```
User Tower:
  user_id       ──→ Embedding ──┐
  gender        ──→ Embedding ──┤
  age           ──→ Embedding ──┤
  occupation    ──→ Embedding ──┤
  zip_code      ──→ Embedding ──┤
  hist_movie_ids──→ Embedding ──→ Mean Pooling ──┤
                                                  ├──→ Concat ──→ MLP [64, 32, 16] ──→ user_vector (16维)
Item Tower:
  movie_id      ──→ Embedding ──┐
  genres        ──→ Embedding ──┤
  year          ──→ Bucketize ──┤
                                  ├──→ Concat ──→ MLP [64, 32, 16] ──→ item_vector (16维)
```

- 激活函数：ReLU
- 输出层不做激活（点积前的向量）
- Loss：sampled softmax（user_vector 与 item_vector 点积后经 softmax 做 N 分类）

### 训练超参建议

| 参数 | 建议值 |
|------|--------|
| embedding_dim | 16 |
| 历史序列长度 | 50（截断或补齐） |
| 负采样数 | 4~10 |
| optimizer | Adam, lr=1e-3 |
| batch_size | 512 |
| epochs | 10~20 |
